# Version 2.1 ÔÇö EHR Integration (Phase 1: Read-Only FHIR) ÔÇö Design

Date: 2026-06-27
Status: Approved (design), pending implementation plan
Scope: Version 2.1 only. Versions 2.2 (eligibility) and 2.3 (Medicaid enrollment) are
separate spec Ôćĺ plan Ôćĺ build cycles and are out of scope for this document.

## 1. Goal

Give patients read-only access to their EHR record (appointments, clinical notes,
medications) inside the existing Patient Portal, and give clinical staff a read-only EHR
summary inside the admin portal ÔÇö all sourced live from a FHIR R4 API over SMART-on-FHIR
OAuth, with **no PHI from the EHR persisted** in our database.

Because no EHR vendor or sandbox credentials exist yet, 2.1 is built as a **swappable
scaffold**: the complete app-side architecture behind a provider abstraction with a
SANDBOX adapter that returns synthetic FHIR resources. Switching to a real EHR is an
environment-variable change, not a code change.

## 2. Key decisions (locked)

1. **FHIR identity link lives on `Patient`, not `User`.** In this codebase `User` is staff
   RBAC; the portal patient is the separate `Patient` model. The spec's "User" was generic.
2. **Reuse the existing `RefillRequest` model.** No new `MedicationRefillRequest` model. The
   EHR "Request Refill" action creates a `RefillRequest` and reuses the existing
   Provider-notification path. Add one optional field to tie it back to the EHR med.
3. **Staff EHR-summary panel gets a new home.** No admin patient-detail screen exists today,
   so add a minimal `/admin/patients` list + `/admin/patients/[id]` detail (gated
   ADMINISTRATOR + CLINICAL_DIRECTOR) to host the read-only EHR Summary panel and the manual
   FHIR-link control (the latter satisfies spec 2.1.4 #4).
4. **Clinical-note "Download as PDF" uses scoped `window.print()`** ÔÇö no new dependency, no
   serverless risk, PHI stays in the browser. (Deliberately not `@react-pdf/renderer`.)
5. **Token cache uses the existing Upstash REST pattern** from `src/lib/rate-limit.ts`. No
   new SDK. If Upstash is unconfigured, token caching is skipped (a fresh token is fetched
   per request) ÔÇö correctness preserved, no hard dependency.

## 3. Architecture

### 3.1 FHIR module ÔÇö `src/lib/fhir/`

- **`types.ts`** ÔÇö minimal typed shapes for the four resources we read: `FhirAppointment`,
  `FhirDocumentReference`, `FhirMedicationRequest`, `FhirPatient`, plus `FhirBundle<T>` and
  `OperationOutcome`.
- **`client.ts`** ÔÇö `fhirGet<T>(path, accessToken): Promise<T>`:
  - Sets `Authorization: Bearer <token>` and `Accept: application/fhir+json`.
  - On 4xx/5xx throws a typed `FhirError` carrying the parsed `OperationOutcome`.
  - Writes exactly one audit entry per call via the existing `audit()`:
    `action: "fhir.read"`, `entity: "<ResourceType>"`, metadata = `{ patientIdHash, status }`.
    Never logs the response body. `patientIdHash` = sha256 of the FHIR patient id.
- **`auth.ts`** ÔÇö two SMART-on-FHIR flows:
  - **Client-credentials (backend/system)** for staff-initiated reads (admin EHR summary).
  - **PKCE authorization-code (standalone launch)** for a patient authorizing read access to
    their own record (portal "Connect Now").
  - Tokens cached in Upstash (REST, via a small helper mirroring `rate-limit.ts`) with TTL =
    `expires_in`. Never written to Postgres. A valid cached token is always reused.
- **`adapter.ts`** ÔÇö mode switch on `EHR_MODE`:
  - `sandbox` (default): returns synthetic FHIR R4 resources for a fixed set of demo patient
    ids. No network calls. No real PHI possible.
  - `production`: calls the real `EHR_FHIR_*` endpoints via `client.ts` + `auth.ts`.
  - `isSandbox()` helper drives the in-app SANDBOX banner.
- **`sandbox-data.ts`** ÔÇö synthetic Appointment / DocumentReference / MedicationRequest /
  Patient fixtures used by the sandbox adapter.

### 3.2 Data model (Prisma)

Add to `Patient`:
```prisma
fhirPatientId  String?   @unique
fhirLinkStatus String?   // "linked" | "pending" | "unlinked"
fhirLinkedAt   DateTime?
```
Add to `RefillRequest`:
```prisma
sourceFhirMedication String?  // set when the refill was requested from the EHR med list
```
No model stores EHR appointment / note / medication content ÔÇö all fetched on demand.

### 3.3 Patient Portal ÔÇö "My Health"

- New nav item in `src/app/patient-portal/layout.tsx` Ôćĺ `/patient-portal/health`.
- Server component reads `fhirLinkStatus` for the session patient.
  - If not `linked`: render the "Connect your health record [Connect Now]" prompt that starts
    the PKCE flow; on success store `fhirPatientId` + set status `linked` + `fhirLinkedAt`.
  - If `linked`: render three tabs (URL-param or segment based, server-rendered):
    - **Appointments** ÔÇö `Appointment?patient={id}&status=booked&date=ge{today}`. Shows
      date/time, provider, location, service type, status. "Need to reschedule?" CTA opens
      the existing appointment-request form pre-filled. Empty state Ôćĺ "Schedule" button.
    - **Clinical Notes** ÔÇö `DocumentReference?patient={id}&category=clinical-note&_sort=-date&_count=10`.
      Lists date, type, author. Each opens a modal rendering `content.attachment`
      (text/plain rendered as-is; text/html sanitized via existing `sanitize.ts`). "Download
      as PDF" = scoped `window.print()`. Content never cached or stored.
    - **Medications** ÔÇö `MedicationRequest?patient={id}&status=active`. Shows name, dosage,
      instructions, prescriber, last-prescribed date. "Request Refill" Ôćĺ creates
      `RefillRequest` (`sourceFhirMedication` set) + notifies PROVIDER. Disclaimer line beneath.

### 3.4 Admin ÔÇö staff EHR summary

- New `/admin/patients` (list) and `/admin/patients/[id]` (detail), gated to ADMINISTRATOR +
  CLINICAL_DIRECTOR via existing RBAC.
- Detail screen "EHR Summary" panel (uses the **client-credentials** flow, not the patient's
  token): last 3 appointments, active-medication count, most-recent clinical-note date+type.
  Labeled "Sourced from EHR ÔÇö read only. To update records, use the EHR directly."
- Same screen exposes a **manual FHIR-link** control: set/clear a Patient's `fhirPatientId`
  (status `linked`/`unlinked`), written to the audit log.

### 3.5 Compliance artifacts

- **`README_HIPAA.md`** (new): BAA prerequisites for v2.1 **and** v2.2; SANDBOX-mode rules;
  "production EHR/eligibility credentials in Vercel Production env only, never in `.env.local`,
  never accessible to Preview."
- **In-app SANDBOX banner** when `EHR_MODE !== "production"`: shown on the portal "My Health"
  section and the admin EHR summary ÔÇö "EHR integration is running in SANDBOX mode. No real
  patient data is being accessed. Contact your compliance officer before switching to
  production credentials."
- **`.env.example`** additions: `EHR_FHIR_BASE_URL`, `EHR_CLIENT_ID`, `EHR_CLIENT_SECRET`,
  `EHR_OAUTH_TOKEN_URL`, `EHR_FHIR_SCOPE` (e.g. `patient/*.read launch/patient openid`),
  `EHR_MODE` (`sandbox` default), each with a comment that production values live only in the
  Vercel Production environment.

## 4. Data flow

1. Patient opens "My Health". Server reads `Patient.fhirLinkStatus`.
2. If linked, server calls `adapter.getAppointments|getNotes|getMedications(fhirPatientId)`.
   - sandbox Ôćĺ fixtures; production Ôćĺ `auth.getToken()` (cached) then `fhirGet()`.
3. Each `fhirGet` writes an audit row (resource type + patient-id hash + status).
4. Note content rendered in-memory in a modal; never stored. Refill request writes a
   `RefillRequest` row + notifies PROVIDER.
5. Admin EHR summary uses the system (client-credentials) token, same adapter, same audit.

## 5. Error handling

- `FhirError` (4xx/5xx) Ôćĺ portal/admin show a friendly "We couldn't load this from your health
  record right now" state; never surface raw `OperationOutcome` to patients.
- Token fetch failure Ôćĺ same friendly state; logged server-side.
- Upstash unavailable Ôćĺ skip cache, fetch a fresh token (no hard failure).
- Unlinked patient Ôćĺ connect prompt, not an error.

## 6. Testing

- Unit: sandbox adapter returns well-formed resources; `fhirGet` audit side-effect fires with
  hashed id and no body; `FhirError` thrown on non-2xx; token cache TTL honored / skipped when
  Upstash unset.
- Component/route: "My Health" renders connect prompt when unlinked and the three tabs when
  linked; refill action creates a `RefillRequest` + notify; admin summary RBAC-gated.
- Manual: SANDBOX banner visible; PDF print path; no EHR content written to DB (grep/inspect).

## 7. Out of scope (this version)

- Writing back to the EHR (refills stay in our DB only).
- Real vendor-specific FHIR quirks (added when a vendor + sandbox is chosen).
- Caching any clinical content.
- Versions 2.2 and 2.3.
