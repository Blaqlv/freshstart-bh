# Version 2.3 — Medicaid Enrollment Support Module — Design

Date: 2026-06-27
Status: Approved (design), pending implementation plan
Branch: `feat/v2.3-medicaid-enrollment` (off `main`, independent of the v2.1 PR)
Scope: Version 2.3 only. v2.1 (EHR) and v2.2 (eligibility) are separate cycles.

## 1. Goal

An internal admin tool that guides staff/Auxlia consultants through the Ohio Medicaid
enrollment and re-validation workflow with structured checklists, tracks document submissions
and enrollment status per provider, and tracks per-MCO enrollment. Architected **tenant-aware
from the start** so Auxlia can white-label it in v4.0 — without a data-model rewrite.

## 2. Key decisions (locked)

1. **Document uploads:** new admin-side flow that reuses the existing secure **storage +
   virus-scan primitives** (`src/lib/storage.ts`, the scan pipeline, `ACCEPTED_MIME`/`MAX_BYTES`
   from `src/lib/documents.ts`). We do **not** reuse the patient-portal `DocumentUpload`
   component (it is hard-wired to the patient upload action + `PatientDocument`).
2. **Audit:** dual — a per-case `EnrollmentAuditEntry` powers the case "Audit Trail" tab, and
   key actions also mirror to the global immutable `AuditLog` via `audit()` so the Compliance
   Officer's `/admin/audit` view sees them.
3. **Scope:** full v2.3 (2.3.1–2.3.5) in one cycle.
4. **Enums vs strings:** `caseType`, `status`, MCO names, and document/MCO statuses stay as
   `String` (matching the spec's literal sets verbatim); allowed values are centralized as
   `const` unions in `src/lib/medicaid/constants.ts` and validated with `zod` at the boundary.
5. **Multi-tenancy:** foundation only — `tenantId` on every enrollment model + a `Tenant`
   model + filter-by-tenant on every query. No tenant-aware auth / subdomain routing / billing
   this cycle (documented in `README_MULTITENANCY.md` for v4.0).

## 3. Data model (Prisma)

New `Tenant` model; six enrollment models, each carrying `tenantId`:

```prisma
model Tenant {
  id        String   @id @default(cuid())
  name      String
  slug      String   @unique
  isActive  Boolean  @default(true)
  createdAt DateTime @default(now())
}

model MedicaidEnrollmentCase {
  id             String    @id @default(cuid())
  tenantId       String
  providerName   String
  providerNpi    String
  caseType       String    // "initial_enrollment" | "revalidation" | "change_of_ownership"
  status         String    @default("not_started") // not_started|in_progress|submitted|approved|rejected|pending_info
  assignedTo     String?   // User.id
  targetDeadline DateTime?
  ohioMedicaidId String?
  notes          String?
  createdAt      DateTime  @default(now())
  updatedAt      DateTime  @updatedAt
  documents      EnrollmentDocument[]
  checklistItems EnrollmentChecklistItem[]
  mcoEnrollments McoEnrollment[]
  auditEntries   EnrollmentAuditEntry[]
  @@index([tenantId, status])
}

model EnrollmentChecklistItem {
  id          String   @id @default(cuid())
  tenantId    String
  caseId      String
  stepNumber  Int
  title       String
  description String
  isRequired  Boolean  @default(true)
  isComplete  Boolean  @default(false)
  completedAt DateTime?
  completedBy String?  // User.id
  notes       String?
  case        MedicaidEnrollmentCase @relation(fields: [caseId], references: [id], onDelete: Cascade)
  @@index([tenantId, caseId])
}

model EnrollmentDocument {
  id           String   @id @default(cuid())
  tenantId     String
  caseId       String
  documentType String   // NPI_letter|W9|liability_insurance|DBH_cert|CARF_cert|corporate_docs|other
  fileName     String
  mimeType     String
  sizeBytes    Int
  storageKey   String?  // R2 object key; null until scan/store completes
  scanStatus   ScanStatus @default(PENDING) // reuse existing enum
  status       String   @default("uploaded") // uploaded|reviewed|accepted|rejected
  reviewNote   String?
  uploadedAt   DateTime @default(now())
  uploadedBy   String
  case         MedicaidEnrollmentCase @relation(fields: [caseId], references: [id], onDelete: Cascade)
  @@index([tenantId, caseId])
}

model McoEnrollment {
  id          String   @id @default(cuid())
  tenantId    String
  caseId      String
  mcoName     String   // Buckeye|CareSource|Paramount|Molina|Anthem|AmeriHealth|Aetna Better Health
  status      String   @default("not_started") // not_started|submitted|approved|rejected
  submittedAt DateTime?
  approvedAt  DateTime?
  notes       String?
  case        MedicaidEnrollmentCase @relation(fields: [caseId], references: [id], onDelete: Cascade)
  @@index([tenantId, caseId])
}

model EnrollmentAuditEntry {
  id          String   @id @default(cuid())
  tenantId    String
  caseId      String
  action      String
  performedBy String
  performedAt DateTime @default(now())
  notes       String?
  case        MedicaidEnrollmentCase @relation(fields: [caseId], references: [id], onDelete: Cascade)
  @@index([tenantId, caseId])
}
```

Notes: `EnrollmentDocument` extends the spec with `mimeType`/`sizeBytes`/`scanStatus` so it
reuses the existing scan pipeline shape (the spec's `fileUrl` becomes `storageKey`, served via
a signed/streamed admin route rather than a stored URL). `scanStatus` reuses the existing
`ScanStatus` enum.

## 4. Multi-tenant foundation (2.3.5)

- `prisma/seed.ts` upserts a Tenant `{ slug: "fresh-start", name: "Fresh Start Behavioral
  Health" }` (idempotent).
- `src/lib/constants.ts` gains `FRESH_START_TENANT_SLUG = "fresh-start"`.
- `src/lib/medicaid/tenant.ts` exports `getActiveTenantId()` — resolves the active tenant id
  by slug (cached per request). All enrollment queries pass `tenantId` in their `where`/`data`.
- A thin data layer `src/lib/medicaid/cases.ts` wraps Prisma so every read/write is
  tenant-scoped in one place (no raw `db.medicaidEnrollmentCase` calls scattered in pages).
- `README_MULTITENANCY.md` lists v4.0 work: tenant-aware auth/session, per-tenant Neon
  branches, subdomain routing, per-tenant billing.

## 5. RBAC

`src/lib/rbac.ts`: add `enrollment:read` and `enrollment:manage`.
- ADMINISTRATOR: both (via `ALL`).
- COMPLIANCE_OFFICER: both (`read` + `manage`).
- CLINICAL_DIRECTOR: `read` only.
Admin nav entry "Medicaid Enrollment" (`/admin/medicaid-enrollment`) gated on `enrollment:read`.
Read-only users see all data; mutation controls are hidden and every mutating server action
calls `requireCapability("enrollment:manage")`.

## 6. Templates, seeding & case creation (2.3.2 / 2.3.4)

- `src/lib/medicaid/checklist-templates.ts` — `initial_enrollment` (14 steps) and
  `revalidation` (7 steps) Ohio checklists as structured arrays, verbatim from the spec. A
  prominent UI disclaimer: "Review with your Medicaid enrollment specialist before using on
  real cases; ODM updates requirements periodically."
- `src/lib/medicaid/mco.ts` — the 7 MCOs.
- **Create case** (`cases.ts`): writes the case, then instantiates `EnrollmentChecklistItem`
  rows from the template for that `caseType`; if `initial_enrollment`, also auto-seeds 7
  `McoEnrollment` rows at `not_started`. All rows carry `tenantId`. Writes the first
  `EnrollmentAuditEntry` ("case created") + global audit.
- `change_of_ownership` is an allowed `caseType` but the prompt supplies no template for it, so
  such a case starts with **no auto checklist items** (staff add items manually later) and no
  MCO auto-seed. Only `initial_enrollment` auto-seeds the 7 MCO rows.

## 7. UI — `/admin/medicaid-enrollment`

Gated `enrollment:read`. Follows existing admin page patterns (`requireCapability`,
`force-dynamic`, server actions).

**Dashboard:** four summary cards (Total Active, Due This Month, Pending Submission, Approved
This Year); case table (provider, type, status, assignee, deadline, completion % =
completed/total checklist items); filters by status/type/assignee; "New Case" button (form).

**Case detail `/[caseId]`** — tabbed:
- **Overview:** metadata, status badge, assignee, notes field, deadline; **MCO tracker grid**
  (one card per MCO with status badge + notes). Status-change control drives the workflow.
- **Checklist:** numbered items with required/optional badge, complete toggle (records
  `completedAt`/`completedBy`), per-item notes; progress bar "N of M complete".
- **Documents:** admin upload (server action + `useActionState`, reusing storage+scan); each
  doc shows type/filename/uploader/date/status; Compliance Officer accept/reject with note.
- **Audit Trail:** chronological `EnrollmentAuditEntry` list; append-only, no edit/delete UI.

## 8. Status workflow + notifications (2.3.3)

`not_started → in_progress → submitted → approved | rejected | pending_info`. Transitions
validated against an allowed-set in `constants.ts`.
- On **submitted:** email the assignee + ADMINISTRATOR addresses via `notify.ts`
  (`notifyStaff` / `sendEmail`).
- On **approved:** "finalize" = set every required `EnrollmentChecklistItem.isComplete = true`
  (recording `completedAt`/`completedBy` = the acting user) and set every `EnrollmentDocument`
  not already `rejected` to `accepted`; then send a congratulations notification.
Every transition writes an `EnrollmentAuditEntry` + global `audit()`.

## 9. Document handling (2.3.3 detail)

`src/lib/medicaid/documents.ts`:
- `storeEnrollmentDocument(actor, caseId, documentType, file)` — validate MIME/size (reuse
  `ACCEPTED_MIME`/`MAX_BYTES`), run the existing virus scan, `putObject` under
  `enrollment/<caseId>/<docId>`, create the `EnrollmentDocument` row, audit.
- `readEnrollmentDocumentBytes(caseId, docId)` — tenant-scoped fetch + `getObject`, streamed
  by an admin route `/admin/medicaid-enrollment/[caseId]/document/[docId]` (capability-gated).
- `setDocumentReview(docId, "accepted"|"rejected", note)` — Compliance Officer action.

## 10. Error handling

- Storage/scan unavailable → upload fails with a clear admin message; no partial row left
  pointing at a missing object (create the row only after a successful store, or mark
  `storageKey=null` + `scanStatus=PENDING` consistently with `PatientDocument`).
- Invalid status transition → server action rejects and returns a field error; no DB write.
- Email send failure → never blocks the status change (notification is best-effort, logged).
- Cross-tenant access → impossible by construction: the data layer always filters by the
  active tenant id; a `caseId` from another tenant returns not-found.
