# Version 2.2 — Insurance Eligibility Automation — Design

Date: 2026-06-27
Status: Approved (design), pending implementation plan
Branch: `feat/v2.2-eligibility` (off `main`, independent of the v2.1 and v2.3 PRs)
Scope: Version 2.2 only.

## 1. Goal

Add real-time insurance eligibility verification to the public insurance form and the intake
portal, surface results to patients immediately and to staff in a dedicated admin queue, and
let staff manage payer-code mappings — all PHI-safe and built as a swappable scaffold so a real
eligibility vendor is an env-var flip.

## 2. Key decisions (locked)

1. **Swappable scaffold, sandbox default.** `lib/eligibility/` behind a provider abstraction with
   a SANDBOX adapter returning synthetic results. `ELIGIBILITY_MODE=sandbox` default; real vendor
   (Waystar/Availity/Eligible) is configured via `ELIGIBILITY_API_*` env in Production only. Same
   pattern as v2.1 EHR.
2. **Keep `FormSubmission`, add `VerificationAttempt`.** The existing encrypted `FormSubmission`
   stays (staff need name/DOB/member ID to manually verify failed/unknown cases); a new PII-free
   `VerificationAttempt` records the automated result and links to the submission.
3. **Dedicated `/admin/insurance` queue** (not the shared generic submissions queue).
4. **Include 2.2.3** (intake background check) this cycle, using Next 16 `after()` for true
   post-response, non-blocking execution.
5. **No coverage amounts persisted.** Copay/deductible are shown to the patient transiently and
   discarded; `VerificationAttempt` stores only `resultStatus` + `rawResponseHash`. Never member
   ID or DOB.
6. **Reuse `billing:manage`** for the admin queue + payer management (Administrator + Billing
   Staff). No new RBAC capability.

## 3. Eligibility module — `src/lib/eligibility/`

- `types.ts` — the prompt's interfaces verbatim:
  ```ts
  interface EligibilityRequest { patientFirstName; patientLastName; patientDob; memberId; insurancePayerCode; }
  interface EligibilityResult {
    status: 'active' | 'inactive' | 'unknown';
    coverageType: string; effectiveDate: string; terminationDate: string | null;
    copay: string | null; deductible: string | null; outOfPocketMax: string | null;
    rawResponseHash: string; // SHA-256 of the raw response — never the response itself
  }
  ```
- `config.ts` — `eligibilityMode()`/`isSandbox()`; env access (`ELIGIBILITY_API_KEY`, `ELIGIBILITY_API_BASE_URL`,
  `ELIGIBILITY_API_VENDOR`); `hashResponse(raw: string): string` (sha256 hex).
- `client.ts` — production HTTP call to the vendor; maps the vendor response to `EligibilityResult`.
- `sandbox-data.ts` — deterministic synthetic results keyed off `memberId`:
  - last digit odd → `active` (with sample copay/deductible/effectiveDate),
  - last digit even → `inactive`,
  - memberId containing `0000` → throws (simulates an API error),
  - non-numeric/empty → `unknown`.
- `adapter.ts` — `checkEligibility(req: EligibilityRequest): Promise<EligibilityResult>`: sandbox →
  synthetic; production → `client.ts`. Always audits one entry (vendor, payerCode, status — **no
  name/DOB/member ID**) and sets `rawResponseHash`. On a thrown vendor/sandbox error, returns a
  result with `status: 'unknown'` (callers treat unknown/inactive/error identically for the patient).
- `index.ts` — re-exports.

## 4. Data model (Prisma)

```prisma
model InsurancePayer {
  id        String  @id @default(cuid())
  name      String
  payerCode String
  isActive  Boolean @default(true)
  @@index([isActive])
}

model VerificationAttempt {
  id              String   @id @default(cuid())
  submittedAt     DateTime @default(now())
  insurerName     String
  payerCode       String
  resultStatus    String   // "active" | "inactive" | "unknown" | "api_error"
  rawResponseHash String
  staffReviewed   Boolean  @default(false)
  staffReviewedAt DateTime?
  staffReviewedBy String?
  source          String   // "insurance_form" | "intake"
  formSubmissionId String?  // links to the encrypted FormSubmission (form path)
  intakeId        String?  // links to the IntakeSubmission (intake path)
  // NEVER stores member ID or DOB.
  @@index([formSubmissionId])
  @@index([intakeId])
  @@index([resultStatus, staffReviewed])
}
```

Seed `InsurancePayer` from the **actual** `acceptedInsurance` list in `src/lib/site.ts` (~14
grouped entries — not the 22 the prompt assumes). Each gets a placeholder `payerCode` (e.g. a
slug of the name) that staff edit in `/admin/settings/payers` before going live. Seeding is
idempotent (upsert on a stable key).

## 5. Payer lookup — `src/lib/insurance/payers.ts`

- `payerCodeFor(insurerName: string): Promise<string | null>` — active `InsurancePayer` whose name
  matches; null if none/inactive.
- `listActivePayers()` / `listAllPayers()` for the form dropdown + admin screen.
- The public form dropdown reads `listActivePayers()` going forward (replacing the static
  `acceptedInsurance` source for the select), so deactivating a payer removes it from the form.

## 6. Public insurance form (2.2.2)

`submitInsurance` (`src/app/_actions/forms.ts`) — after creating the encrypted `FormSubmission`:
1. The form has a single `name` field; split it best-effort into first/last (first token =
   first name, remainder = last name) for the `EligibilityRequest`. `payerCodeFor(provider)`; if
   found, `checkEligibility({first,last,dob,memberId,payerCode})`.
2. Write a `VerificationAttempt` (`source:"insurance_form"`, `formSubmissionId`, `resultStatus`,
   `rawResponseHash`). Map a thrown/unknown to `resultStatus:"api_error"`/`"unknown"`.
3. Return a non-PHI result object to the client (`{ status, coverageType?, effectiveDate?, copay?,
   deductible? }`).
`InsuranceVerificationForm` renders the mandated banner from that result:
- `active` → green: "Your [Insurance] coverage appears active as of [effectiveDate]. A staff member
  will confirm your benefits before your appointment." + copay/deductible if present.
- otherwise → yellow: "We were unable to verify your coverage automatically. A staff member will
  contact you within 1 business day to assist. You can also call us at 937-579-0073."
- Raw API errors are never shown.
Existing honeypot, rate-limit (3/IP/hr), encryption, public-submission log, and `notifyStaff`
behavior all stay.

## 7. Admin (2.2.2 #5 + 2.2.4)

- **`/admin/insurance`** (gated `billing:manage`): list insurance `FormSubmission`s with their
  latest `VerificationAttempt` (result status, payer, time). `active` rows show as auto-verified;
  a staff action toggles `staffReviewed` / flags manual review (records `staffReviewedBy/At`). Link
  to the existing encrypted submission detail for follow-up.
- **`/admin/settings/payers`** (gated `billing:manage`): `InsurancePayer` CRUD — edit `payerCode`,
  toggle `isActive`, add a payer. Every mutation writes the global `audit()` log.
- An `EligibilitySandboxBanner` (renders only when `isSandbox()` is true) on both screens.

## 8. Intake background check (2.2.3)

- `saveStep` (`src/app/intake/actions.ts`): when the saved step is the `insurance` step, call
  `after(() => runIntakeEligibility(intakeId))` (from `next/server`) so it runs after the response
  — the wizard advances with no added latency.
- `runIntakeEligibility(intakeId)` (in `src/lib/eligibility/intake.ts`): decrypts the intake data,
  reads name/dob/insuranceProvider/memberId, resolves the payer code, runs `checkEligibility`, and
  writes a `VerificationAttempt` (`source:"intake"`, `intakeId`). Best-effort: never throws into
  the request.
- **Confirmation** (`/intake/complete`): read the latest `VerificationAttempt` for the intake →
  "Your insurance coverage was verified automatically." (active) or "We will verify your insurance
  before your appointment." (otherwise / none).
- **Admin intake review** (`/admin/intake/[id]`): show the latest eligibility result for the intake.

## 9. Compliance & RBAC

- `.env.example`: `ELIGIBILITY_MODE="sandbox"`, `ELIGIBILITY_API_KEY`, `ELIGIBILITY_API_BASE_URL`,
  `ELIGIBILITY_API_VENDOR`, with production-env-only guidance.
- `README_HIPAA.md`: create with the eligibility BAA/sandbox section (and the EHR section, since
  this branch is off `main`). If the v2.1 PR merges first, resolve the additive conflict.
- No new RBAC capability; `billing:manage` already belongs to Administrator + Billing Staff.

## 10. Error handling

- Vendor/sandbox throw → `checkEligibility` returns `status:"unknown"`; callers record
  `"unknown"`/`"api_error"` and show the yellow banner. Patient never sees a raw error.
- Payer not found for the selected insurer → skip the API call, record `resultStatus:"unknown"`,
  yellow banner.
- `after()` job failure in intake → swallowed (logged); the intake is unaffected.
- Eligibility config missing in production mode → `checkEligibility` throws internally and is
  caught → `unknown` (no crash, no PHI leak).

## 11. Testing

`tsx` scripts:
- `eligibility-sandbox.mjs` — determinism (odd memberId active, even inactive, `0000` → error path
  yields `unknown`, empty → unknown); `rawResponseHash` matches `/^[0-9a-f]{64}$/`; the returned
  `EligibilityResult` has no `memberId`/`dob` keys.
- `eligibility-config.mjs` — `isSandbox()` default true; `hashResponse` stable + hex.
- `payers.mjs` (pure helper portion) — name normalization/match logic if extracted as a pure fn.
UI/route/intake wiring verified via `npm run build` + scoped `eslint`.
